# 架构方法论：全能SOUL + 上下文项目拆分

## 为什么不用多Agent分发

传统做法是模拟公司组织架构：设一个"总Agent"调度，下面挂多个"专家Agent"分管不同领域。这种模式有严重问题：

1. **信息在Agent间传递会丢失** — 总Agent转述给专家Agent时，上下文不可避免地被压缩和歪曲
2. **多了一层分发判断** — 用户问个问题，先过总Agent判断分给谁，再由专家处理，链路变长
3. **跨领域问题处理不好** — 现实中的问题往往跨多个领域，多Agent模式下要协调好几个角色
4. **本质错误** — 人类分部门分岗位是因为一个人脑子装不下所有知识。AI没有这个限制，一个Agent完全可以精通所有领域

> 模拟公司设置Agent协同，是把AI的优点丢了，换上了人类的弱点。

## 全能SOUL是什么

一个SOUL.md = 一个全知全能的助手。它同时精通用户需要的所有领域，不需要"开会""传话""对齐信息"。

SOUL.md采用分层结构：
- **第一层：人设与核心原则**（~300字）— 角色定位、沟通风格、行为准则
- **第二层：领域方法论精华**（每领域200-300字）— 核心思维框架和要点，不含具体模板
- **第三层：4大自动工作流**（~800字）— 建项/结项/归档/资料消化的操作步骤
- **第四层：模板索引**（~200字）— templates/目录映射，按需读取

总量控制在2500字以内。具体模板存放在templates/中按需读取，不塞进SOUL.md。

## 上下文项目拆分是什么

不按"职能领域"组织工作目录，按"要办的事"组织。

### 对比

| 按职能分（旧） | 按事项分（新） |
|---|---|
| docs/finance/ docs/okr/ docs/recruiting/ | projects/Q3-OKR-Review/ projects/张三试用期/ |
| 问"张三试用期不过要不要换人"→ 同时涉及培养+招聘+OKR | 所有相关信息在一个项目目录里，天然打通 |
| 找信息要翻多个目录 | 一个目录就是一件事的完整上下文 |

### 项目目录标准结构

```
{项目名}/
├── context.md    # 上下文：背景、目标、进展、关键决策
└── output/       # 产出物：文档、报表、PPT等
```

### 项目的生命周期

```
用户说"帮我搞XX" → AI自动在projects/建目录 → 从templates/拷模板 → 从memory/读数据
        ↓
      工作中... → 持续更新context.md和output/
        ↓
用户说"这事结了" → AI移入archive/ → 关键摘要写入memory/MEMORY.md
```

## 模板库的定位

templates/不是"专家Agent"，而是"知识增强层"：

- SOUL.md内嵌方法论精华（如OKR评分标准的核心原则）
- templates/存放可直接使用的完整模板文件（如OKR制定的具体表格）
- 建项时AI自动从templates/拷贝合适的模板到项目目录

新增领域：在templates/加目录放模板，在SOUL.md方法论中补一段描述即可。

## 长期记忆的作用

memory/MEMORY.md 持续积累跨项目的核心数据：
- 人员信息、预算数据、进度快照等持久化事实
- 每个项目结项时，AI提取关键摘要写入memory
- 新建项目时，AI从memory读取相关历史数据填入context.md

这保证了跨项目的知识连续性——即使一个项目结了，它的经验教训仍然可以被新项目利用。
